# QR 웹 자동 로그인 후속 작업

현재 웹 쪽 기본 흐름(티켓 생성/승인/소비/세션 발급)은 구현됨.
아래는 앱과 DB(및 운영)에서 추가로 해야 할 작업 목록.

## 1) 모바일 앱 작업

- 딥링크 수신 처리 구현
  - 예: `bookjuk://qr-login?token=...`
  - 앱이 실행 중/백그라운드/종료 상태 모두에서 토큰 파싱 가능해야 함.

- QR 스캔 화면 연동
  - 카메라 스캔 결과에서 토큰 추출
  - 추출 실패 시 사용자 안내(유효하지 않은 QR, 만료 QR 등)

- 로그인 사용자 컨텍스트로 승인 호출
  - 앱 로그인 사용자의 `users_id`를 기준으로 `approveQrLoginTicket` 호출
  - 현재는 웹 데모 버튼으로 승인하고 있으므로, 실제 앱에서 해당 호출로 대체 필요

- 보안 UX 보강
  - 최소한의 디바이스 인증(PIN/생체인증) 후 승인 여부 검토
  - 오승인 방지를 위한 간단한 확인 문구/버튼 고려

- 에러 케이스 UX
  - 만료됨, 이미 사용됨, 네트워크 오류, 권한 오류를 분리 안내
  - 재시도/QR 재요청 동선 제공

## 2) DB / Supabase 작업

- 스키마 적용
  - `supabase/qr_login_schema.sql`를 실제 프로젝트 DB에 실행
  - `login_tickets`, `web_sessions` 테이블/인덱스 생성 확인

- RLS 정책 추가 (필수)
  - `login_tickets`, `web_sessions`에 대해 RLS enable
  - 웹 클라이언트가 직접 광범위 조회/수정하지 못하게 제한
  - 권장: 클라이언트 직접 업데이트 대신 RPC/Edge Function 경유

- 승인/소비 로직의 서버 사이드 이전 권장
  - 현재 로직 일부가 프론트에 있으므로, 최종 운영은 Edge Function 또는 서버 API에서 처리
  - 트랜잭션으로 원자성 보장:
    - `approved -> used` 전이
    - 세션 생성
    - 중복 소비 방지

- 만료 데이터 정리 작업
  - 주기적으로 `expired` 처리 또는 오래된 레코드 삭제(cron)
  - 운영 규모에 따라 보관 정책(예: 7일/30일) 수립

- 레이트 리밋/감사 로그
  - 티켓 생성, 승인, 소비 API에 rate limit
  - 최소 로그 필드: timestamp, ticket_id, users_id, user_agent/ip(가능 시), 결과 코드

## 3) 웹 쪽 후속 개선

- 폴링 -> Realtime 전환 검토
  - 현재 1.5초 폴링 구현
  - 트래픽 증가 시 Supabase Realtime 구독으로 전환 고려

- 세션 저장소 하드닝
  - 현재는 `localStorage` 기반
  - 최종 운영은 `HttpOnly Secure SameSite` 쿠키 기반 세션 권장

- 로그인 화면 정리
  - 지금은 개발/검증 목적의 "테스트 즉시 승인" 버튼 포함
  - 운영 배포 전 제거 필요

## 4) 테스트 체크리스트 (추가)

- 정상 시나리오
  - 티켓 생성 -> 앱 승인 -> 웹 자동 로그인 완료

- 실패 시나리오
  - 만료 티켓 승인
  - 이미 사용된 티켓 재소비
  - 빈/손상 토큰 입력
  - 권한 없는 사용자 승인 시도

- 보안 시나리오
  - 토큰 재전송(replay) 방지 확인
  - 짧은 TTL 내/외 동작 검증
  - 동시 요청 경쟁 상태에서 단일 성공 보장
